Please create a safe QA / test-validation strategy for the new public eSIM activation state flow so we can verify all user-facing order states without waiting for real backend provisioning to happen naturally.

Main objective:
We have already implemented the public activation-state UX in the Orders page. The processing state is visible and looks correct. Now we need a reliable way to validate the other states:

- processing
- ready to activate
- active
- expired
- error / support needed

We do NOT want to break production logic or fake the real customer flow permanently. We need a controlled way to test and validate each state.

Core goal:
Make it possible to inspect and verify every public order state safely, using controlled test data or dev-only tooling.

Critical constraints:

1. Preserve production behavior
- Do not break the current real order flow
- Do not alter production checkout behavior
- Do not alter real provisioning logic in production
- Do not break auth, admin, admin orders, or backend order-management logic

2. No unsafe hacks in production
- Do not leave dangerous public state-switch controls exposed to normal users
- Any test tooling must be dev-only, admin-only, local/staging-only, or otherwise safely gated
- If using seeded test orders, clearly isolate them from real orders where possible

3. Preserve current state-derivation logic
- Do not rewrite the activation-state UI unnecessarily
- The goal is to validate the existing state logic against controlled inputs

PHASE 1 — Audit current state inputs

First inspect the current code and data model used to derive the public activation states.

Please identify:
- which fields currently drive the user-facing activation state
- whether the UI state is derived from fields such as:
  - status
  - payment_status
  - esim_delivered_at
  - qr_code / qrCode / qr_url / similar
  - activated_at / expires_at
  - provisioning status / delivery status
- where this logic currently lives in code

Please summarize the current activation state derivation before adding test support.

PHASE 2 — Choose the safest validation approach

Implement one safe validation strategy, preferring reuse of existing test/staging patterns if present.

Preferred options, in order of priority:

Option A — Seeded test orders in dev/staging
Create a small set of seeded test orders representing each activation state.

Option B — Dev-only state override
Add a dev-only/test-only mechanism to override the displayed activation state for a known test order.

Option C — Admin-only state preview tool
If appropriate, create an admin-only preview mechanism for state validation.

Important:
Do NOT expose state manipulation to normal public users.

PHASE 3 — Create validation coverage for these states

We need to be able to verify these five states end-to-end in the Orders UI:

1. Processing
Expected UI:
- PT: "Ativação em preparação"
- waiting message shown
- no QR code
- no download button
- no top-up
- support CTA visible

2. Ready to activate
Expected UI:
- PT: "Pronto para ativação"
- green readiness block
- QR visible (if real field exists)
- download QR / eSIM button visible
- activation guide visible
- email reassurance visible

3. Active
Expected UI:
- PT: "Ativo"
- active-state message visible
- QR view available if appropriate
- top-up available only if supported
- activation guide still accessible if intended

4. Expired
Expected UI:
- PT: "Expirado"
- expired messaging visible
- buy new plan CTA visible
- top-up/renew only if truly supported

5. Error / needs support
Expected UI:
- PT: "Precisa de apoio"
- error/support message visible
- support CTA visible
- no misleading ready/active actions
